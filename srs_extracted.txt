System Architecture1.1 Architecture StyleThe platform is designed as a Modular Monolith — a single deployable unit with strictly bounded internal modules. This choice optimises for speed-to-market, operational simplicity, and team coherence at launch while preserving clean module boundaries that allow future extraction into independent services.Design RationaleAdmin-led assignment is the operational default at launch. Auto-dispatch, real-time tracking, and multi-city dispatch engines are designed as future drop-in replacements inside the same schema and API surface.ConcernDecisionBackend runtimeNode.js 20 LTS + Express 5ORM / MigrationsPrisma 5 (schema-first, type-safe)Primary databasePostgreSQL 15 via Neon (serverless-friendly)Object storageAWS S3 (primary) or Cloudinary (fallback/images)WhatsApp messagingMeta Cloud API or Twilio WhatsApp BusinessWeb clientReact 18 + Vite + Tailwind CSSStaff / customer appReact Native (Expo managed workflow)Background jobsBullMQ + Redis 7 (notification worker, reminders)AuthJWT (RS256) + Refresh Token rotationPaymentsStripe (webhook + idempotency keys) — Phase 2CDNCloudFront in front of S3 for public assetsSecrets managementAWS Secrets Manager / environment .env vault1.2 Component MapWeb Client (React/Vite) — Customer booking UI, Admin Operations ConsoleMobile Client (React Native / Expo) — Customer bookings + Staff visit executionREST API Server (Express) — All business logic, RBAC, domain rulesBackground Worker (BullMQ) — Async notifications, renewal reminders, cleanupPostgreSQL (Neon) — Primary relational data store with full ACID guaranteesRedis — BullMQ job queue, rate-limit counters, idempotency key storeS3 / CloudFront — Medical documents, staff documents, visit imagesWhatsApp Provider — Family notifications, booking acknowledgementsStripe — In-app payments and subscription billing (Phase 2)1.3 Module Directory Structuresrc/  modules/    auth/          # No business logic.ServiceOrchestrate business rules, call repositories, manage transactions, enqueue jobs.RepositoryPrisma data access only. No business logic. Returns domain models.MiddlewareAuth, RBAC, request validation (Zod), audit logging, error normalisation.WorkerAsynchronous side-effects: WhatsApp delivery, renewal reminders, cleanup.1.5 Deployment ModelSingle API process at launch, stateless and horizontally scalable from day 1Separate Worker process (can run as sibling container or separate Render/Railway service)Shared PostgreSQL and Redis accessible by both API and WorkerS3 accessed directly from API (presign) and uploaded-to by clientsAPI instances behind AWS ALB or Railway load balancer for zero-downtime deploysGitHub Actions CI/CD pipeline: lint → test → build → deploy to staging → prod gateSECTION 2 — DATA MODELS (PRISMA SCHEMA)2. Concurrency & Race Condition HandlingWhy This MattersAt moderate booking volume, two admins can simultaneously assign the same staff member to overlapping visits, or two customers can simultaneously cancel/reschedule the same booking. Without explicit locking, these races produce silent data corruption. The following patterns must be implemented before launch.5.1 Staff Assignment Race — Optimistic Locking + SELECT FOR UPDATEWhen an admin assigns a staff member to a visit, the following atomic transaction must execute:// src/modules/assignments/assignment.service.tsasync assignStaffToVisit(visitId: string, staffUserId: string, adminUserId: string) {  return prisma.$transaction(async (tx) => {    // 1.
